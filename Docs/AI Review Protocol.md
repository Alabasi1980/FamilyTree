# AI Review Protocol

## 1. الغرض

يحدد هذا الملف القواعد الإلزامية لأي مساعد ذكاء اصطناعي يقوم بمراجعة مشروع **FamilyTree / بستان الأصول**.

الهدف هو منع:

* مراجعة نسخة قديمة من المشروع.
* إصدار أحكام غير مدعومة بالأدلة.
* الخلط بين ما هو موجود فعليًا وما هو متوقع.
* الادعاء بوجود أو عدم وجود ميزة دون فحص ملفاتها.
* تقديم تقييم عام قبل استكمال الحد الأدنى من المراجعة.

هذا الملف هو المرجع الأول قبل أي تحليل أو اقتراح أو تقييم للمشروع.

---

# 2. المستودع المرجعي

المستودع الرسمي:

```text
https://github.com/Alabasi1980/FamilyTree
```

يجب عدم افتراض أن أي مستودع قديم أو رابط معاد توجيهه يحتوي على أحدث نسخة.

يجب التحقق فعليًا من:

* اسم المستودع.
* الفرع المطلوب.
* آخر Commit.
* SHA الكامل للـ Commit.
* تاريخ آخر تحديث إن كان متاحًا.

---

# 3. قاعدة إلزامية قبل بدء المراجعة

قبل تقديم أي رأي أو تحليل، يجب على المراجع عرض البيانات التالية:

```text
Repository:
Branch:
Commit SHA:
Commit name:
Review started from:
```

لا تعتبر المراجعة صالحة إذا لم يتم تثبيت Commit محدد.

عند عدم القدرة على معرفة آخر Commit، يجب التصريح بذلك صراحة وعدم الادعاء بأن النسخة المراجعة هي الأحدث.

---

# 4. مصدر الحقيقة

ترتيب مصادر الحقيقة هو:

1. الكود الموجود في الـ Commit المحدد.
2. ملفات قاعدة البيانات والـ migrations.
3. إعدادات المشروع.
4. الاختبارات ونتائج البناء.
5. وثائق المشروع المعتمدة.
6. وصف المستخدم.

عند وجود تعارض بين الذاكرة السابقة والكود الحالي، يكون الكود الحالي هو المرجع.

عند وجود تعارض بين وثيقة قديمة والكود الحالي، يجب توضيح التعارض وعدم اختيار أحدهما دون دليل.

---

# 5. منع الاعتماد على مراجعات سابقة

لا يجوز استخدام حكم سابق على المشروع دون إعادة التحقق من النسخة الحالية.

أمثلة على أحكام ممنوعة دون تحقق جديد:

* المشروع ما زال يستخدم Mock Data.
* لا توجد قاعدة بيانات.
* لا توجد لوحة إدارة.
* لا توجد مصادقة.
* التطبيق جاهز للإطلاق.
* التطبيق غير جاهز.
* هذه الميزة غير موجودة.
* هذه المكتبة مستخدمة.
* هذا المسار لا يعمل.

كل حكم يجب أن يكون مرتبطًا بملف أو نتيجة اختبار.

---

# 6. تصنيف المعلومات

يجب تصنيف كل نتيجة مهمة إلى واحدة من الحالات التالية:

## مؤكد

معلومة موجودة صراحة في الكود أو الإعدادات أو قاعدة البيانات.

الصيغة المطلوبة:

```text
الحالة: مؤكد
الدليل: path/to/file
التفصيل: ...
```

## استنتاج

نتيجة منطقية مبنية على أكثر من دليل، لكنها ليست مكتوبة صراحة.

الصيغة المطلوبة:

```text
الحالة: استنتاج
الأدلة:
- path/to/file
- path/to/another/file
الاستنتاج: ...
```

## غير متحقق

معلومة لم يتم فحص ملفاتها أو اختبارها بما يكفي.

الصيغة المطلوبة:

```text
الحالة: غير متحقق
السبب: ...
المطلوب للتحقق: ...
```

يمنع تحويل الاستنتاج إلى حقيقة مؤكدة.

---

# 7. الأدلة المطلوبة

أي ملاحظة جوهرية يجب أن تتضمن:

* مسار الملف.
* اسم المكوّن أو الدالة أو الجدول عند الحاجة.
* رقم السطر إن كانت الأداة توفره.
* Commit SHA الذي تمت القراءة منه.
* نتيجة الاختبار إن كانت الملاحظة متعلقة بالتشغيل.

مثال صحيح:

```text
الحالة: مؤكد
الملف: prisma/schema.prisma
الدليل: يوجد model باسم Person مرتبط بـ Family.
```

مثال غير مقبول:

```text
يبدو أن التطبيق يستخدم قاعدة بيانات حقيقية.
```

# 9. مراجعة آخر التحديثات

عند طلب مراجعة أحدث نسخة، يجب:

1. معرفة آخر Commit.
2. تثبيت SHA الكامل.
3. مراجعة Diff الخاص به.
4. تحديد الملفات المضافة والمعدلة والمحذوفة.
5. عدم الاكتفاء برسالة الـ Commit.
6. قراءة الملفات الحالية بعد التعديل، وليس الـ Diff فقط.

رسالة الـ Commit ليست دليلًا كافيًا على اكتمال الميزة.

---

# 10. تشغيل المشروع

عند توفر بيئة تشغيل، يجب تنفيذ ما أمكن من:

```bash
npm install
npm run build
npm run lint
npm test
```

أو الأوامر الفعلية الموجودة في `package.json`.

يجب تسجيل:

* الأمر.
* النتيجة.
* الأخطاء.
* التحذيرات.
* ما لم يتم تشغيله وسبب ذلك.

لا يجوز القول إن المشروع يعمل لمجرد أن الكود يبدو صحيحًا.

---

# 11. تقييم الجاهزية

يمنع استخدام عبارة:

```text
التطبيق جاهز
```

دون تحديد نوع الجاهزية.

يجب الفصل بين:

* جاهزية التصميم.
* جاهزية البروتوتايب.
* جاهزية التطوير الداخلي.
* جاهزية الاختبار.
* جاهزية النسخة التجريبية.
* جاهزية الإنتاج.
* جاهزية النشر على المتجر.

أي نسبة جاهزية يجب أن تكون مبنية على معايير واضحة، لا على الانطباع.

---

# 12. طريقة عرض النتائج

يجب أن يبدأ تقرير المراجعة بهذه المعلومات:

```text
Repository:
Branch:
Commit SHA:
Commit name:
Files reviewed:
Commands executed:
Areas not reviewed:
```

ثم تعرض النتائج بالترتيب:

1. مشاكل حرجة.
2. مشاكل عالية الأولوية.
3. مشاكل متوسطة.
4. تحسينات اختيارية.
5. نقاط القوة المؤكدة.
6. الأمور غير المتحقق منها.

---

# 13. درجات الخطورة

## Critical

مشكلة قد تسبب:

* فقدان بيانات.
* تسريب بيانات.
* تجاوز صلاحيات.
* توقف التطبيق.
* فساد علاقات شجرة العائلة.
* فشل النشر.

## High

مشكلة تؤثر على وظيفة أساسية أو تجربة شريحة كبيرة من المستخدمين.

## Medium

مشكلة لها بديل أو تأثير محدود، لكنها تحتاج إصلاحًا.

## Low

تحسين جودة أو تنظيم أو تجربة، ولا يمنع الاستخدام.

---

# 14. قواعد الصراحة

يجب استخدام العبارات التالية عند الحاجة:

```text
لم أتحقق من ذلك.
لا أملك دليلًا كافيًا.
هذا استنتاج وليس حقيقة مؤكدة.
لم أقرأ الملف المسؤول عن هذه الميزة بعد.
لم أتمكن من تشغيل المشروع.
لا أستطيع تأكيد أن هذا هو آخر Commit.
```

يمنع ملء الفراغ بتوقعات أو معلومات محتملة.

---

# 15. منع المجاملة والموافقة التلقائية

لا يجوز للمراجع:

* تأييد رأي المستخدم دون فحص.
* معارضة رأي المستخدم دون دليل.
* تغيير الحكم فقط لأن المستخدم اعترض.
* استخدام لغة واثقة لتعويض نقص المعلومات.

المطلوب هو الحكم المستند إلى الأدلة، حتى لو خالف توقع المستخدم أو المراجع السابق.

---

# 16. قاعدة إعادة المراجعة

إذا تبين أن المراجعة تمت على Commit خاطئ أو نسخة قديمة:

1. تعتبر جميع الأحكام السابقة غير صالحة.
2. يجب إعلان ذلك بوضوح.
3. تبدأ المراجعة من الصفر.
4. يمنع إعادة استخدام النتائج القديمة دون تحقق.
5. يجب ذكر سبب الخطأ وكيف تم تصحيحه.

---

# 17. التغييرات على المستودع

لا يجوز تعديل أي ملف أو إنشاء Commit أو Branch أو Pull Request دون طلب صريح من مالك المشروع.

المراجعة الافتراضية هي قراءة فقط.

قبل أي تعديل يجب توضيح:

* الملفات التي ستتغير.
* سبب التعديل.
* المخاطر.
* الفرع المستهدف.
* طريقة التراجع.

---

# 18. البيانات الحساسة

يمنع عرض أو نسخ:

* كلمات المرور.
* مفاتيح API.
* أسرار البيئة.
* رموز المصادقة.
* بيانات المستخدمين الخاصة.
* معلومات الاتصال الشخصية.
* محتوى قواعد البيانات الفعلي.

عند اكتشاف سر داخل المستودع، يجب الإشارة إلى وجود المشكلة دون إعادة نشر السر.

---

# 19. شرط قبول المراجعة

تعتبر المراجعة مقبولة فقط إذا:

* تم تثبيت Commit محدد.
* تم ذكر الملفات التي تمت قراءتها.
* تم دعم النتائج بالأدلة.
* تم فصل الحقائق عن الاستنتاجات.
* تم ذكر ما لم يتم التحقق منه.
* لم تصدر أحكام عامة مبكرة.
* لم يتم الاعتماد على ذاكرة محادثة سابقة.

---

# 20. تعليمات بدء أي جلسة جديدة

عند بدء محادثة جديدة، يجب إعطاء المساعد النص التالي:

```text
اقرأ ملف AI_REVIEW_PROTOCOL.md أولًا والتزم به.
تحقق من آخر Commit في مستودع Alabasi1980/FamilyTree.
لا تبدأ التحليل قبل أن تعرض اسم الـ Commit وSHA الكامل.
لا تستخدم نتائج مراجعات سابقة دون إعادة التحقق.
```

---

# 21. قاعدة نهائية

الثقة لا تبنى على أسلوب الكتابة أو درجة التأكد في الإجابة.

الثقة تبنى فقط على:

```text
نسخة محددة
+ ملفات محددة
+ أدلة قابلة للتحقق
+ اختبارات فعلية
+ تصريح واضح بحدود المعرفة
```
